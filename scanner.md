# Scanner Structure (scanner.cpp)

The `scanner.cpp` file is a lexical analyzer generated by **Flex** from the `scanner.l` definition file. It contains the logic required to break down an input stream into a sequence of tokens.

## Core Components

### 1. `yylex()`

This is the most critical function in the scanner. It is the main entry point for lexical analysis.

- **Purpose**: It reads the input stream and matches characters against the regular expressions defined in `scanner.l`.
- **Operation**: It uses a generated state machine (DFA) to efficiently find the longest match. Once a match is found, it executes the corresponding C++ action (e.g., calling `emit()`).
- **Return Value**: It typically returns the type of the matched token or `0` (YY_NULL) when the end of the input is reached.

### 2. `yyin` and `yyout`

- **`yyin`**: A `FILE*` pointer that determines where the scanner reads its input from. By default, it points to `stdin`.
- **`yyout`**: A `FILE*` pointer for the scanner's output (used by the `ECHO` macro). By default, it points to `stdout`.

### 3. `emit(TokenType t, const char* text)`

Although defined in the user section of `scanner.l`, this function is compiled into `scanner.cpp`.

- **Purpose**: It records each recognized token.
- **Action**: It creates a `Token` object, adds it to the global `g_tokens` vector, and prints the token type (and lexeme for IDs/literals) to the console.

## Buffer Management Functions

Flex provides several functions to manage the input stream:

- **`yyrestart(FILE* new_file)`**: Tells the scanner to start reading from a new file. It flushes the internal buffers and resets the scanner's state.
- **`yy_create_buffer(FILE* file, int size)`**: Allocates and initializes a new input buffer state for a given file.
- **`yy_switch_to_buffer(YY_BUFFER_STATE new_buffer)`**: Switches the scanner's active input buffer to a previously created one.
- **`yy_delete_buffer(YY_BUFFER_STATE buffer)`**: Frees the memory associated with an input buffer.

## Internal State Tables

The file contains several large arrays (e.g., `yy_accept`, `yy_ec`, `yy_nxt`, `yy_chk`). These tables represent the **DFA (Deterministic Finite Automaton)** generated by Flex to perform high-speed pattern matching.
